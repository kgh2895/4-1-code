## 답안 작성 행동 지침

Andrej Karpathy의 LLM 코딩 원칙을 답안 작성에 적용한 가이드라인.  
**트레이드오프:** 이 지침은 속도보다 신중함을 우선한다. 단순한 작업은 판단에 따른다.

### 1. 작성 전 사고

**가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

답안 작성 전에:
- 문제 조건을 명시적으로 진술한다. 불확실하면 묻는다.
- 여러 해석이 가능하면 제시한다 — 조용히 고르지 않는다.
- 더 단순한 설명이 있으면 말한다. 필요할 때 이의를 제기한다.
- 불명확하면 멈춘다. 무엇이 혼란스러운지 명명하고 묻는다.

### 2. 단순함 우선

**문제를 해결하는 최소 설명. 추측성 내용 금지.**

- 문제에서 요구하지 않은 내용은 추가하지 않는다.
- 핵심 원리를 먼저, 세부 사항은 그 다음.
- 일어날 수 없는 시나리오에 대한 추론은 포함하지 않는다.
- 답안이 과도하게 길면 핵심 논리만 남겨 다시 쓴다.

스스로 묻기: "채점자가 이걸 보고 핵심이 명확하다고 할까?" 그렇지 않으면 정리한다.

### 3. 외과적 수정

**필요한 것만 건드린다. 기존 노트를 임의로 수정하지 않는다.**

기존 노트 편집 시:
- 요청받지 않은 차시의 내용은 수정하지 않는다.
- 기존 평가의견과 제출 답변은 보존한다.
- 새 차시 내용 추가 시 기존 구조(details 태그, 구분선)를 따른다.

### 4. 목표 주도 실행

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

- "답안 작성" → 평가 기준(루브릭)에서 각 항목이 커버되는지 확인 후 제출
- "내용 보완" → 평가의견의 지적 사항을 직접 대응하는 내용인지 확인
- 다단계 작업은 간단한 계획을 먼저 진술한다

### H-모빌리티.md 작성 규칙

- 차시별로 `<details><summary>N차시 - 제목</summary>` 구조 유지
- 문제 답안은 `**제출 답변**` 헤더 아래 작성
- 평가의견은 `**평가의견**` 헤더 아래 유지 (수정 금지)
- 학습 목표, 핵심 개념 정리는 각 차시 `<details>` 안에 작성

## Cross-Verify (요청 시 실행)

사용자가 "검증해줘", "cross-verify", "크로스체크", "다른 모델 의견" 등을 요청하면  `/cross-verify` 스킬을 실행한다.

실행 방식:
1. Gemini (gemini-2.5-pro)
2. Codex (gpt-5.5, xhigh reasoning)
3. Gemini + Codex 병렬 (권장)
